# 003 - 0.2.1 dependency-probe isolation and supply-chain hardening

- Status: Accepted
- Date: 2026-07-02
- Supersedes: none (extends ADR 002)

## Context

A follow-up world-class review of `@naughtyjs/proberesponder` after the 0.2.0
hardening pass focused on three areas: (1) resilience of the dependency
prober against misbehaving custom `Prober` implementations, (2) supply-chain
and CI security posture, and (3) substantiating the package's "high
performance" positioning with a reproducible benchmark instead of an
unverified claim.

Two additional reliability hypotheses were investigated and **disproved**
empirically (via Docker across Node 20/26 and raw-socket reproductions)
before any code was written, specifically to avoid introducing speculative
"defensive" code for problems that do not actually exist:

- An unhandled `ECONNRESET` while writing an HTTP probe response does **not**
  crash the process — Node's `http` module already attaches its own internal
  socket error handler regardless of user code.
- An un-drained POST body hitting a 404'd route on a keep-alive connection
  does **not** corrupt the next pipelined request — Node's `http` module
  auto-drains the request body before reusing the connection.

Both are documented here so the reasoning is not lost and this ground isn't
re-tread by a future reviewer.

## Decision

### Dependency-probe batch isolation

`probeDependencies()` used `Promise.all` over one promise per `Prober`, with
only the `check()` call guarded by try/catch. A custom `Prober` whose
`serviceId()` or `affectsStatuses()` implementation throws was _outside_ that
guard, so `Promise.all` would reject and **the entire cycle's results for
every other prober in the batch were silently discarded** — a single broken
third-party `Prober` could blank out every dependency status for that cycle.

Fix: every prober's `serviceId()`, `affectsStatuses()`, and `check()` calls
now run inside a single try/catch per prober (`probeSingle`). A failure at
any stage marks that prober `NOT OK` and, if identity could not be
determined, falls back to a synthetic `prober[<index>]` id with no affected
statuses (see "Alternatives considered" for the fail-closed-per-entry vs.
fail-open-for-others trade-off). Every other prober in the batch is
completely unaffected, and the internal `Promise.all` can now never reject.

### Pluggable logger — considered, declined for this round

Considered adding an injectable `Logger` (error/warn) to replace the
hardcoded `console.error`/`console.warn` call sites, to integrate with
structured logging (pino/winston) commonly used by the Kubernetes services
this package targets.

Declined for this pass: it is a legitimate idea for a future minor release,
but it is the one true API-surface expansion that was on the table, and it
enlarges the review surface without fixing a demonstrated failure the way
the batch-isolation bug above does. Revisit if/when requested.

### CI and supply-chain hardening

- `ci.yml` did not declare a `permissions:` block (unlike `release.yml`,
  which already scoped `contents: read` at the top level and
  `packages: write` only on the publish job). Added top-level
  `permissions: contents: read` to `ci.yml` — it only checks out code and
  runs local checks, so read-only is correct least-privilege.
- Added `.github/workflows/codeql.yml`: CodeQL static analysis for
  `javascript-typescript`, triggered on push/PR to `main` plus a weekly
  schedule, `build-mode: none` (JS/TS analysis needs no compilation). Scoped
  to `contents: read` + `security-events: write` only — narrower than
  GitHub's own default template, which additionally grants `packages: read`
  and `actions: read` for cases (private repos, external CodeQL packs) that
  do not apply here. Verified against GitHub's official documentation and
  the `actions/starter-workflows` template rather than assumed.
- `github/codeql-action` is pinned to a commit SHA with a `# v4` comment,
  matching this repo's existing convention for `actions/checkout` and
  `actions/setup-node`, and is already covered by the existing
  `github-actions` Dependabot ecosystem entry — no `dependabot.yml` changes
  needed.

### esbuild advisory (GHSA-g7r4-m6w7-qqqr) — risk accepted

`npm audit` reports one low-severity transitive advisory: esbuild's own dev
server allows arbitrary file reads on Windows. `tsup@8.5.1` (latest) pins
`esbuild: ^0.27.0`, which cannot resolve to the patched `0.28.1` without a
manual `overrides` entry forcing a pre-1.0 esbuild version bump that tsup has
not validated against.

Accepted without a forced override because:

- Severity is low, and the vector is specific to esbuild's own local dev
  server (`esbuild --serve` / `esbuild.serve()`), which neither `tsup`
  (library-mode bundler, not a dev server) nor this project's tooling ever
  invokes.
- The advisory is Windows-only; CI runs exclusively on `ubuntu-latest`.
- Forcing an untested pre-1.0 esbuild minor bump (0.27.x -> 0.28.x, a
  breaking-change boundary under esbuild's own versioning policy) into the
  build pipeline is a larger correctness risk than the advisory itself.
- Dependabot already tracks this transitively and will pick up a compatible
  `tsup`/`vite` release automatically once one exists.

Revisit if `npm audit` severity increases, or a compatible upstream release
becomes available.

### Benchmark

Added `scripts/bench.mjs` (zero new dependencies; uses only `node:http`,
`node:perf_hooks`, `node:timers/promises`) and an `npm run bench` script,
exercising:

1. `contentNegotiator` in isolation (pure-function throughput).
2. A full HTTP request/response round trip against the real `createServer()`
   server over loopback with a keep-alive client agent, reporting req/sec
   and p50/p95/p99/max latency.

Documented example output in the README with explicit caveats that numbers
are illustrative and hardware/load-dependent, plus instructions to reproduce
locally. Not wired into CI: shared CI runners are not a reliable environment
for latency/throughput numbers and would only add flakiness.

### Deferred: major Dependabot updates

Five open Dependabot PRs bump major versions (`eslint` 9->10, `typescript`
5->6, `@types/node` 22->26, `actions/checkout` 4->7, `actions/setup-node`
4->6). Left untouched in this pass: each can carry breaking changes to lint
rules, type-checking behavior, or Action input/output contracts, and
deserves its own focused review and CI-green verification rather than being
bundled into an unrelated resilience/hardening change.

## Alternatives considered

1. Fail-open (mark all of Startup/Ready/Live affected) when a Prober's
   `affectsStatuses()` throws.
    - Pros: never hides a problem behind an unaffected status.
    - Cons: a broken _metadata_ method in one third-party Prober plugin
      could take down startup/readiness/liveness for the entire process — a
      much bigger blast radius than the bug being fixed, and a new,
      opinionated failure-propagation behavior nobody asked for.
    - Decision: fail-closed to `[]`; the failure is still fully visible in
      `healthResponse()` under a synthetic id.

2. Force the esbuild override via `package.json` `overrides`.
    - Pros: clears the `npm audit` finding immediately.
    - Cons: esbuild pre-1.0 minor bumps are breaking-change boundaries;
      untested against `tsup`/`vite`'s actual usage; the vulnerability's
      preconditions (Windows, dev server) don't apply here anyway.
    - Decision: accept and document; let it flow through the normal
      Dependabot cadence.

3. Ship the pluggable logger now alongside the resilience fix.
    - Pros: bigger single release, addresses a real integration gap.
    - Cons: expands API surface and review scope for what is otherwise a
      bug-fix release; better reviewed on its own.
    - Decision: defer.

## Consequences

Positive:

- A single misbehaving custom `Prober` can no longer erase every other
  dependency's status for an entire probing cycle.
- CI and CodeQL workflows follow least-privilege token permissions.
- Ongoing static security analysis via CodeQL, in addition to `npm audit`.
- Performance claims are now backed by a runnable, documented benchmark
  instead of assertion alone.
- Two previously-unverified reliability assumptions were empirically
  confirmed as non-issues, avoiding speculative/unnecessary defensive code.

Negative:

- `DependencyStatus.serviceId` can now be a synthetic `prober[<index>]`
  value instead of always reflecting the real `Prober`'s intended id, in the
  narrow case where `serviceId()` itself throws.
- The low-severity esbuild advisory remains open (tracked, not fixed) until
  upstream (`tsup`/`vite`) ships a compatible release.
